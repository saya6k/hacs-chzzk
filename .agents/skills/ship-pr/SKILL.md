# Ship a change (commit → push → PR → merge)

release-drafter's autolabeler reads the **PR title** to label it, then its
version-resolver picks the bump from that label. A malformed PR title falls
back to a patch bump instead of the intended one. Defer to
[[conventional-commit]] for wording and [[hacs-preflight]] for sanity checks.

## 1. Branch

- **Never commit on `main`.** `git switch -c <type>/<short-slug>`
  (e.g. `feat/add-rain-sensor`, `fix/auth-retry`).

## 2. Pre-PR checks

- Run [[hacs-preflight]] (Python compile, JSON, manifest keys, i18n parity,
  brand asset). CI repeats HACS + hassfest but does not replace local checks.

## 3. Commit

- Use [[conventional-commit]]. **Never** `--no-verify` / `--no-gpg-sign`.
- Don't hand-bump `manifest.json` `version` — the `sync-manifest-version` CI
  job owns it.

## 4. Push & open PR

- `git push -u origin <branch>`.
- `gh pr create` — the **PR title MUST be a valid Conventional Commit**. Its
  `pull_request_target` trigger runs release-drafter's autolabeler against
  this title, so a malformed title mislabels the PR (defaults to patch).

## 5. Merge

- This is a **single package**, so a **squash merge is fine** — the PR title is
  the one release-relevant commit. (No multi-scope rebase concern as in
  `ha-apps`.)
- After merge, release-drafter updates its **draft** GitHub Release with the
  new entry, and `sync-manifest-version` pushes the bumped `manifest.json`
  straight to `main` (commit tagged `[skip ci]`).

## 6. After merge

- `git switch main && git pull --ff-only origin main` — picks up the
  `sync-manifest-version` commit.
- **Publishing the draft release** (manual, in GitHub UI) tags the release and
  triggers `docs.yml` to deploy the Zensical site. HACS surfaces the new
  release to users once published.

## IMPORTANT

- The draft release body is auto-generated by release-drafter — edit it in
  the GitHub UI before publishing if it needs cleanup, don't hand-edit
  `.github/release-drafter.yml`'s template per-release.
- Set the repo's **Pages source to "GitHub Actions"** once, so the Docs
  workflow can publish the Zensical site.
